  - ETL (Extract, Transform, Load) or ELT (Extract, Load, Transform)
    - get data from one data store to another,
    - typically involves three main steps:
      - Extract: 
        - The process of retrieving data from various source systems,
      - Transform: 
        - The process of cleaning, 
          - enriching, and 
          - transforming the data into a format suitable for analysis and reporting,
        - common transformations include: 
          - union, 
          - join, 
          - filter, 
          - sort, 
          - aggregate, 
          - pivot, 
          - unpivot,  
          - regex, and

            - Data Cleaning
            - Data Enrichment
            - Data Aggregation
            - Data Filtering
            - Data Sorting
            - Data Joining
            - Data Splitting
            - Data Validation
              - prevent errors at source
              - ensure accuracy and consistency of data
              - manual or automated rules
            - Data Standardization
            - Data Normalization
            - Data Denormalization
            - Data Masking
            - Data Anonymization
            - Data Encryption
      - Load: The process of loading the transformed data into a target data store, such as a data warehouse or data lake.

  - Blockchain
    - distributed ledger technology that allows for secure and transparent transactions 
    - without the need for a central authority(broker or intermediary)
      - like banks or governments or Amazon Marketplace
    - used in various industries, including finance, supply chain management, healthcare, and more.
    - Key features of blockchain include:
      - Decentralization: Blockchain operates on a decentralized network of computers, 
        - eliminating the need for a central authority to manage transactions.
      - Transparency: All transactions on the blockchain are visible to all participants, 
        - ensuring transparency and accountability.
      - Security: Blockchain uses cryptographic techniques to secure transactions and 
        - prevent unauthorized access or tampering.
      - Immutability: Once a transaction is recorded on the blockchain, 
        - it cannot be altered or deleted, ensuring the integrity of the data.
      - Consensus Mechanism: Blockchain relies on consensus mechanisms, 
        - such as Proof of Work (PoW) or 
        - Proof of Stake (PoS), 
        - to validate and confirm transactions on the network.
    - Common use cases for blockchain include:
      - Cryptocurrencies: Bitcoin, Ethereum, and other cryptocurrencies 
        - are built on blockchain technology, 
        - allowing for secure and decentralized digital currency transactions.
      - Supply Chain Management: Blockchain can be used to track and verify the movement of goods 
        - through the supply chain, ensuring transparency and reducing fraud.
    - transaction can be:
      - peer-to-peer transactions,
      - smart contracts,
      - decentralized applications (DApps),
      - identity management,
      - voting systems,
      - healthcare records management,
      - real estate transactions,
      - intellectual property protection,
      - financial services,
      - and more.
  - Smart contracts
    - self-executing contracts with the terms of the agreement directly written into code. 
    - run on blockchain platforms, such as Ethereum, and 
    - automatically execute when predefined conditions are met.
    - Key features of smart contracts include:
      - Automation: Smart contracts automatically execute when predefined conditions are met, 
        - eliminating the need for intermediaries and reducing the risk of human error.
      - Transparency: The terms of the contract are visible to all parties involved, 
        - ensuring transparency and accountability.
      - Security: Smart contracts use cryptographic techniques to secure transactions and 
        - prevent unauthorized access or tampering.
      - Immutability: Once a smart contract is deployed on the blockchain, 
        - it cannot be altered or deleted, ensuring the integrity of the agreement.
    - Common use cases for smart contracts include:
      - Financial Services: Smart contracts can be used for various financial applications, 
        - such as decentralized finance (DeFi) platforms, 
        - insurance claims processing, and 
        - cross-border payments.
      - Supply Chain Management: Smart contracts can automate and verify transactions in the supply chain, 
        - ensuring transparency and reducing fraud.
      - Real Estate: Smart contracts can facilitate property transactions, 
        - automating processes such as title transfers and rental agreements.
      - Healthcare: Smart contracts can be used to manage healthcare records, 
        - automate insurance claims, and facilitate secure data sharing between healthcare providers.
      - Identity Management: Smart contracts can be used to manage digital identities, 
        - allowing individuals to control their personal information and securely share it with trusted parties.
      - Voting Systems: Smart contracts can be used to create secure and transparent voting systems, 
        - ensuring the integrity of the voting process and reducing the risk of fraud.
      - Intellectual Property Protection: Smart contracts can be used to manage intellectual property rights, 
        - automate royalty payments, and facilitate secure licensing agreements.

  - Data Warehousing
    - process of collecting, storing, and managing large volumes of data from various sources to support business intelligence and analytics activities. 
    - Data warehousing involves the extraction, transformation, and loading (ETL) of data from operational systems into a central repository, 
    - where it can be organized, optimized, and made available for analysis and reporting.
    - Key components of a data warehouse include:
      - Data Sources: Operational systems, external data sources, and other databases that provide the raw data for the data warehouse.
      - ETL Process: The process of extracting data from source systems, transforming it into a consistent format, and loading it into the data warehouse.
      - Data Storage: The central repository where the data is stored, typically organized in a star schema or snowflake schema for efficient querying and analysis.
      - Metadata: Information about the data, such as its source, structure, and usage, which helps users understand and navigate the data warehouse.
      - Data Access Tools: Tools and interfaces that allow users to query, analyze, and visualize the data in the data warehouse, such as SQL clients, business intelligence tools, and data visualization platforms.
    - Common use cases for data warehousing include:
      - Business Intelligence: Data warehouses provide a centralized repository of data that can be used for reporting, analysis, and decision-making activities.
      - Data Analytics: Data warehouses enable organizations to perform complex analytics and gain insights from their data, such as customer behavior analysis, sales forecasting, and market research.
      - Data Integration: Data warehouses can integrate data from multiple sources, providing a unified view of the organization's data assets and enabling cross-functional analysis.
      - Data Governance: 
        - Data warehouses can support data governance initiatives by providing a centralized repository 
        - for data management, quality control, and compliance activities.
      - Data Archiving: 
        - Data warehouses can serve as a long-term storage solution for historical data, 
        - allowing organizations to retain and access data for regulatory compliance, auditing,

    - DAO: Data Access Object
      - design pattern that provides an abstract interface to a database or other persistence mechanism. 
      - encapsulates the logic for accessing and manipulating data, 
      - allowing for separation of concerns and improved maintainability of the application code.
      - typically includes methods for performing CRUD (Create, Read, Update, Delete) operations on the underlying data store, 
      - as well as methods for managing database connections and transactions.
      - Benefits of using a DAO include:
        - Abstraction: DAOs provide a layer of abstraction between the application code and the underlying data store, allowing for flexibility and ease of maintenance.
        - Reusability: DAOs can be reused across different parts of the application, reducing code duplication and improving consistency.
        - Testability: DAOs can be easily tested in isolation, allowing for better unit testing and improved code quality.
      - SQL can be stored in form of:
        - Embedded SQL in application code
          - dedicated class as DAO (Data Access Object) or Repository
          - needs to be compiled and deployed with the application code
          - e.g. CustomerDAO.java, OrderDAO.java, ProductDAO.java, etc.
          - needs downtime for changes and updates
          - Prone to SQL injection attacks
          - Dynamic SQL
          - Network latency, as the SQL code is executed on the application server and requires communication with the database server, which can lead to slower performance compared to stored procedures.
        - Stored Procedures
          - Dedicated by function or business logic
          - Executable on the database server(RDBMS)
          - e.g. CustomerDAO, OrderDAO, ProductDAO, etc.
          - can be called from application code or other stored procedures
          - no need to compile and deploy with the application code, 
          - as they are stored and executed on the database server
          - no downtime
          - Less prone to SQL injection attacks, 
            - as the SQL code is stored and executed on the database server, 
            - rather than being embedded in the application code.
          - Not dynamic, as they are precompiled and stored on the database server, 
            - which can limit their flexibility and adaptability to changing business requirements.
          - preferred method: access database objects through stored procedures, 
            - as it provides better security, maintainability, and performance compared to embedding SQL in application code. 
            - By using stored procedures, you can centralize your database logic, 
              - making it easier to manage and update as needed, 
              - without requiring changes to the application code.
          - offers layer of abstraction between the application code and the database, 
            - allowing for better separation of concerns and improved maintainability of the application code. 
            - By encapsulating the database logic within stored procedures, you can reduce the risk of SQL injection attacks and improve the overall security of your application. 
            - Additionally, stored procedures can be optimized for performance, 
              - as they are precompiled and stored on the database server, 
              - which can lead to faster execution times compared to embedding SQL in application code.
          - processed on the database server, 
            - which can reduce network latency and improve performance compared to embedding SQL in application code, 
            - as the SQL code is executed directly on the database server, 
            - rather than requiring communication between the application server and the database server. 
            - This can lead to faster execution times and improved overall performance of your application.

  - DataOps
    - DataOps is a set of practices and tools that aim to improve the speed, quality, and reliability of data analytics and data science projects by applying principles from DevOps to the data lifecycle. 
    - DataOps focuses on collaboration, automation, and continuous improvement across the entire data pipeline, from data ingestion and processing to analysis and visualization.
    - Key principles of DataOps include:
      - Collaboration: DataOps emphasizes collaboration between data engineers, data scientists, analysts, and other stakeholders to ensure that everyone is aligned on project goals, requirements, and timelines.
      - Automation: DataOps promotes the use of automation tools and processes to streamline data workflows, reduce manual errors, and increase efficiency in data processing and analysis.
      - Continuous Integration and Continuous Deployment (CI/CD): DataOps encourages the use of CI/CD practices to enable rapid development, testing, and deployment of data pipelines and analytics solutions.
      - Monitoring and Feedback: DataOps emphasizes the importance of monitoring data pipelines and analytics solutions in real-time to identify issues early and provide feedback for continuous improvement.
      - Version Control: DataOps promotes the use of version control systems for managing code, configurations, and documentation related to data projects to ensure reproducibility and traceability.
      - Data Quality Management: DataOps emphasizes the importance of maintaining high data quality through automated testing, validation, and monitoring processes throughout the data lifecycle.
      - Scalability: DataOps encourages the design of scalable data architectures and workflows that can handle increasing volumes of data as projects grow.
      - Security and Compliance: DataOps promotes the implementation of security best practices and compliance measures to protect sensitive data throughout the data lifecycle.
      - Data Governance: DataOps emphasizes the importance of data governance practices to ensure that data is properly managed, documented, and accessible to the right stakeholders while maintaining data privacy and security.
      - Continuous Learning and Improvement: DataOps encourages a culture of continuous learning and improvement, where teams regularly review their processes, tools, and outcomes to identify areas for enhancement and optimize their data workflows over time.
    - DataOps Practices:
      - Automation

  - Change and Release Management
    - Change Management: The process of managing changes to data systems, processes, and workflows to ensure that changes are implemented smoothly and with minimal disruption to business operations.
    - Release Management: The process of planning, scheduling, and controlling the deployment of changes to data systems, processes, and workflows to ensure that releases are delivered on time and with minimal risk to business operations.
    - Key components of Change and Release Management include:
      - Change Request Management: The process of submitting, reviewing, and approving change requests for data systems, processes, and workflows, including impact assessment and risk analysis.
      - Change Implementation: The process of implementing approved changes to data systems, processes, and workflows, including testing, validation, and documentation of changes.
      - Release Planning: The process of planning and scheduling releases of changes to data systems, processes, and workflows, including coordination with stakeholders and communication of release schedules.
      - Release Deployment: The process of deploying approved changes to data systems, processes, and workflows, including monitoring and validation of releases to ensure successful deployment and minimal disruption to business operations.
      - Change and Release Documentation: The process of documenting changes and releases to data systems, processes, and workflows, including change logs, release notes, and post-implementation reviews to capture lessons learned and improve future change and release management processes.
      - Change and Release Communication: The process of communicating changes and releases to stakeholders, including end-users, management, and other relevant parties, to ensure awareness and understanding of changes and releases and to manage expectations and minimize resistance to change.
      - Change and Release Governance: The process of establishing governance structures and policies for managing changes and releases to data systems, processes, and workflows, including roles and responsibilities, approval processes, and compliance requirements to ensure that changes and releases are managed effectively and in accordance with organizational policies and standards.
      - Change and Release Metrics: The process of tracking and analyzing metrics related to change and release management, such as change success rates, release frequency, and post-release incident rates, to evaluate the effectiveness of change and release management processes and identify areas for improvement.
      - Change and Release Automation: The process of automating change and release management processes, such as change request submission, approval workflows, testing, and deployment, to increase efficiency and reduce manual errors in change and release management.
      - Change and Release Training: The process of providing training and education to stakeholders involved in change and release management, including change request submitters, approvers, implementers, and end-users, to ensure that they understand their roles and responsibilities in the change and release management process and are equipped with the necessary skills and knowledge to effectively manage changes and releases to data systems, processes, and workflows.
      - Rollback and Recovery: The process of planning and executing rollback procedures in case of failed changes or releases, including backup and recovery strategies to minimize downtime and data loss in the event of a failed change or release.
    - Types of changes:
      - Data source changes
      - Model or transformation changes
      - System configuration changes
      - Security and access control changes
      - Architectural changes
      - Process and workflow changes
      - Tool and technology changes
      - Compliance and regulatory changes
      - Performance optimization changes
      - Bug fixes and issue resolution changes
      - User interface and experience changes
      - Data quality improvement changes
      - Data governance and documentation changes
      - Data retention and archiving changes
      - Data integration and interoperability changes
      - Data visualization and reporting changes
      - Data science and machine learning changes
      - DataOps and automation changes
      - Change and release management process changes
    - Performing a release:
      - Minimize impact downstream by scheduling releases during low-usage periods or implementing phased rollouts to gradually introduce changes and mitigate potential disruptions to business operations.
      - Communicate release schedules and potential impacts to stakeholders in advance to manage expectations and ensure awareness of upcoming changes, allowing for proper planning and preparation.
      - Implement robust testing and validation processes to ensure that changes are thoroughly tested and validated before deployment, reducing the risk of issues and minimizing the likelihood of post-release incidents that could impact business operations.
      - Establish clear rollback procedures and backup strategies to quickly revert changes in case of issues during or after deployment, minimizing downtime and data loss in the event of a failed release and ensuring business continuity.
      - System Stability: Ensure that the release does not introduce instability to the system, which could lead to performance degradation or downtime, by thoroughly testing and validating changes before deployment and closely monitoring the system after release.
